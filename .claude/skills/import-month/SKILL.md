# Импорт месячной вкладки Google Sheets в прод

Инструмент: `scripts/import-sheets.mjs` (dry-run по умолчанию, `--post <baseUrl>` пишет).
Прод доступен только через port-forward:

```bash
kubectl --context lab port-forward deploy/money-counter -n money-counter 18787:8787 &
# ... работа через http://localhost:18787 ...
# в конце: kill %1 / lsof -ti :18787 | xargs kill
```

## Главный принцип

**Каждый CSV — потенциально другая «версия приложения».** Формат листа уже менялся
(в апреле 2026 в таблицу добавили колонку, и суммы переехали из G в H), колонки
переводов используются нестандартно, формулы листа в CSV-экспорт не попадают.
Поэтому: никогда не `--post` без пре-скана и dry-run; любое новое предупреждение
или неожиданный ноль в счётчиках — повод разобраться в структуре файла, а не
пропустить.

## Шаг 0 — файлы

Лежат в `~/Downloads`, имена содержат пробелы и варьируются
(`2025 - ... Money counter - 2026-05.csv`, `2025_Money_counter_2026-04.csv`) —
глоби аккуратно: `"$HOME/Downloads/"*"Money counter"*"2026-06.csv"`.

## Шаг 1 — пре-скан (обязательный)

Прочитай файл глазами/скриптом ДО dry-run:

1. **Раскладка колонок.** Известные версии:
   | Версия | Сумма («Чек») | EUR-эквивалент | Потери перевода |
   |---|---|---|---|
   | до 2026-04 | G (idx 6) | K (idx 10) | M (idx 12) |
   | 2026-04+   | H (idx 7) | L (idx 11) | N (idx 13) |

   `detectLayout()` в скрипте выбирает по тому, где реально числа. Если суммы
   не в G и не в H — это НОВАЯ версия: расширь `detectLayout`, не заливай вслепую.

   **Дом. валюта листа (важно для ранних 2025).** Колонка «EUR-эквивалент» на
   самом деле хранит сумму в ДОМАШНЕЙ валюте листа. Пока пользователь жил в
   Таиланде (по ~март/апрель 2025) дом. валюта = **THB**, поэтому эта колонка =
   THB-эквивалент, а НЕ EUR. Признак: у THB-счетов (Kasikorn/Krungsri) idx6 ==
   idx10 (нет конвертации), у не-THB-счетов idx10 = idx6 × ~34. Для таких вкладок
   добавляй флаг **`--home-thb`** (см. шаг 2/3) — иначе снятия и наличные лягут с
   батовыми числами в евро. С 2025-05 дом. валюта уже EUR — флаг не нужен.

   **Вкладки 2024 (книга «2023 - 2024»)** — раскладка G/M, дом. валюта THB,
   формат строк тот же (даты «чт 04.07.2024», «➕/➖ <счёт>»). Для них нужен
   ЕЩЁ и флаг **`--cash-thb`** («глубокий Таиланд»: вся наличность — баты):
   точное «Снятие наличных» с THB-счёта кладёт ногу прихода на **CASH THB той
   же суммой** (это баты→баты, НЕ кросс-валютный случай 2025-01..03), а строки
   «➖ наличные» уходят на CASH THB даже при ПУСТОЙ колонке эквивалента
   (прецеденты 2024-09/11: у пары строк формула эквивалента потерялась при
   экспорте). НЕ включай `--cash-thb` для вкладок 2025-01..03 — там снятые баты
   менялись на евро и ноги CASH EUR заведены вручную.
2. **Строки-переводы** (`E` начинается с «На …»): выпиши те, у которых заполнены
   «лишние» колонки (I/J/K при H-раскладке). Скрипт считает приход как
   `сумма − потери`; если в потерях лежит не то (прецедент: «Kast → Ledger,
   21,02 − 12,82 = 8,20» — загадочные комиссии), приход получится странным.
   Такие строки НЕ чинить эвристиками — импортировать как есть и **пометить
   флажком** после заливки (см. шаг 5).
3. **Снятия наличных.** Описание ровно «Снятие наличных» → скрипт сам создаст
   приход на CASH EUR из колонки EUR-эквивалента. Нечёткие упоминания
   («Всякое (снятие наличных)») скрипт только перечислит в отчёте — решает
   пользователь (прецедент: такую строку пользователь осознанно НЕ зачислял).
   При `--home-thb` снятие с не-EUR счёта = кросс-валютное (THB→EUR): EUR-суммы
   в CSV нет, поэтому скрипт заливает ТОЛЬКО расход и просит добавить/связать
   ногу CASH EUR вручную (прецедент 2025-03: Kasikorn 11 213,94 ฿, Krungsri
   7 482,55 ฿ — помечены флажком). Наличная строка без конвертации (idx6==idx10)
   при `--home-thb` уходит на **CASH THB** (баты), а не CASH EUR.
4. **«На <не-счёт>»** (dtac, Билайн …) — это оплата услуг, не перевод; скрипт
   корректно импортирует расходом и покажет в отчёте с пометкой «(не счёт → расход)».

## Шаг 2 — dry-run

```bash
node scripts/import-sheets.mjs "<file.csv>" [ещё файлы]
# ранние 2025 (дом. валюта THB): добавь --home-thb
# вкладки 2024: --home-thb --cash-thb
```

Проверь в отчёте:
- строку `[раскладка …]` — совпадает с пре-сканом (при `--home-thb` там же
  «дом. валюта THB»);
- `⚠ НЕИЗВЕСТНЫЕ счета` — НЕ выдумывай маппинг: спроси пользователя имя счёта в
  приложении и валюту, добавь в `ACCOUNTS` в скрипте (маппинг идёт по
  name+currency; «Kast» без суффикса = KAST VD; VITA — виртуальный, пропускается);
- суммы по счетам против шапки СЧЕТА в самом листе (порядок величин);
- счётчики переводов/снятий/предупреждений — любое новое ⚠ разобрать.

## Шаг 3 — заливка

```bash
node scripts/import-sheets.mjs "<file.csv>" --post http://localhost:18787
# ранние 2025: ...--home-thb --post http://localhost:18787
```

Отчёт чанка: `+N оп, +M счетов, дублей D, отклонено R`.
- `+M счетов` должно быть **0** для уже существующих счетов. Ненулевое = маппинг
  разошёлся с именем в приложении (прецедент: слал `CASH` вместо `CASH EUR`/
  `CASH THB` — route создал бы дубль). Проверь `ACCOUNTS` и удали дубль.
- `+M счетов` — импорт автосоздаёт недостающие счета (opening balance 0, без дат
  жизни). После заливки напомни пользователю выставить начальный остаток и дату
  открытия в Настройки → Счета.
- Отклонённые печатаются с причинами. «вне периода существования счёта» — у счёта
  в приложении стоят даты открытия/закрытия, конфликтующие с выпиской: решить с
  пользователем (подвинуть даты или строки не нужны).

**Идемпотентность**: повторный `--post` безопасен (дедуп по
`счёт|дата|сумма|описание`). Но строки, созданные пользователем вручную с ДРУГИМ
описанием, дедупом не ловятся — появится дубль (прецедент: ручная нога
«Поступление» против канонического «Снятие наличных» → задвоившиеся +220).
Если после заливки появился неожиданный `+1..2` — найди дубль, удали его и
переименуй ручную строку в каноническое описание.

## Шаг 4 — верификация

Через API (тот же port-forward) сверь с dry-run:

```bash
curl -s "http://localhost:18787/api/periods"
curl -s "http://localhost:18787/api/transactions?from=2026-06-01&to=2026-06-30&limit=500" | jq '.transactions | length'
```

Количество операций месяца должно совпасть с числом из dry-run. Помни про
февраль (28/29) и 30-дневные месяцы — никогда не подставляй «-31» механически.
Вкладка ≠ календарный месяц: вкладка месяца N захватывает 1–2 первых числа
месяца N+1, поэтому сверяй по датам строк, а не по файлам.

**Золотая сверка балансами (исторические вкладки).** В шапке каждой вкладки
блок «Тек. показатели» — это балансы счетов на КОНЕЦ вкладки (последнее
редактирование), и они образуют точную цепь между месяцами:
`конец(M) = конец(M−1) + нетто строк вкладки M` (проверено на 2024-07..12 до
копейки). После заливки сверяй `GET /api/accounts?asOf=<дата начала следующей
вкладки>` с этими цифрами. Нюансы листа: строка «Krungsri» в шапке = банк +
отдельный депозит 319 592,44 ฿ одной суммой; у наличных есть строка
расхождения («не сходится», к концу 2024-12 = −3 579,75 ฿), поэтому цифра
наличных в шапке чуть расходится с суммой строк — это свойство листа, ничего
не досоздавать.

**Начальные остатки НЕ трогать.** Досыпка старой истории сдвигает сегодняшние
балансы (баланс = opening + сумма операций) — это ОЖИДАЕМО, пользователь
выставляет opening сам. Правка opening — не часть импорта: не менять и не
создавать корректировочные операции без явной просьбы (фидбек 2026-07-23:
самовольное перебазирование после импорта 2024-07..12 пришлось откатывать).
В отчёте достаточно НАЗВАТЬ сдвиг по счетам. Если пользователь попросит
поправить: `PATCH /api/accounts?id=N {"openingBalance": <units>}`; значение
идёт через `parseMoneyInputToCents`, число трактуется как ЕДИНИЦЫ валюты
(×100) — передавай единицы, не центы. Справочно из цепочки балансов листа:
перед 2024-07-04 было Kasikorn 660 219,15 / Krungsri 319 592,44 (депозит) /
CASH THB 1 540,25.

## Шаг 5 — пост-обработка

1. **Флажки** на подозрительные строки из пре-скана (п.2): найди id через
   `GET /api/transactions?from=<дата>&to=<дата>`, затем
   `PATCH /api/transactions?id=N` c `{"flagged":true,"notes":"Импорт: <что проверить>"}`.
2. Свяжи переводы: `GET /api/transfers/detect` → просмотри пары глазами
   (детектор консервативный, но проверь) → `POST /api/transfers {"pairs":
   [{"outId":…,"inId":…},…]}`. Либо скажи пользователю нажать **«Найти
   переводы»** в приложении. Кросс-валютные переводы связываются вручную
   карандашом → тип «Перемещение».
3. Перечисли в ответе: созданные счета, помеченные строки, нечёткие «снятия»,
   всё отклонённое.

## Санитарные правила

- Никаких записей в прод без просмотренного dry-run.
- Тестовые записи в проде не создавать; проверки — read-only (GET) либо на
  локальном dev (`npm run dev`, порт 3000+).
- Изменения в `scripts/import-sheets.mjs` (новая раскладка, новый маппинг) —
  закоммитить с объяснением, какой месяц её принёс.
