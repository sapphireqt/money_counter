# 1. Контекст продукта

Family Finance — семейное приложение для:

- учёта операций по нескольким банковским и наличным счетам;
- работы с разными валютами;
- просмотра актуальных балансов;
- аналитики расходов по категориям;
- учёта переводов между своими счетами;
- прогнозирования;

MVP уже существует. Основные пользователи — одна семья. Наиболее частый сценарий — ручное внесение операций задним числом и последовательная обработка периодов. 
Приоритет текущей задачи — **desktop**.

# 2. Главные продуктовые инварианты

Эти правила обязательны во всех компонентах, а не только в таблице.

## 2.1. Отображение денежных сумм

- Расходная операция показывается **без минуса**.
- Поступление показывается **зелёным цветом и с плюсом**.
- Минус может отображаться у отрицательного баланса счёта.
- Внутри модели расход может храниться отрицательным значением — это не должно протекать в UI.
- Для денежных значений используй tabular numerals.
- Не дублируй валюту там, где она уже однозначно определяется контекстом, но не жертвуй ясностью.

## 2.2. Визуальные ограничения

- Не использовать логотипы банков и счетов.
- Не добавлять декоративные цветные маркеры без смысловой необходимости.
- Основная иерархия строится типографикой, выравниванием, расстояниями и очень лёгкими разделителями.
- Не превращать каждую строку или настройку в отдельную карточку.
- Не использовать служебные или технические термины в пользовательском интерфейсе, если они не помогают принять решение.

## 2.3. Работа с датой и счётом

- Порядок полей `Счёт` и `Дата` должен быть одинаковым во всех формах.
- Дата операции влияет на список доступных счетов.
- При изменении даты нужно проверять, был ли выбранный счёт активен в эту дату.
- Нельзя молча сохранить операцию в счёт, который ещё не существовал или уже был закрыт.
- Для перевода датой операции считается дата списания.

# 3. Что не переносить из прототипа в production

В прототипе есть служебная панель состояний. Она нужна только для QA и сравнения вариантов.

Не выводить обычному пользователю:

- панель `Состояния`;
- переключатели mock-сценариев;
- искусственные loading/error/empty-переключатели;
- переключение `Таблица Б`;
- демонстрационные варианты форм;
- import-модалки и import-state из prototype;
- фиктивные операции и счета.

`Таблица Б` может остаться только как development/Storybook/QA-вариант. Production desktop использует **Таблицу А: отдельная колонка счёта**.

Если в проекте есть Storybook, route для playground или dev-only feature flags, альтернативные состояния можно перенести туда. В пользовательский build они попадать не должны.

# 4. Единая сетка desktop-экрана

Используй CSS variables или эквивалентные токены, чтобы все горизонтальные блоки подчинялись одной сетке.

Целевые значения из прототипа:

```css
--sidebar-w: 72px;

--left-col: 922px;
--right-col: 350px;
--content-gap: 18px;
--content-max: 1346px; /* включает горизонтальные padding по 28 px */
```

Sidebar в рамках текущего этапа **всегда свёрнут**. Не реализуй expanded-состояние, toggle расширения и отдельную геометрию для широкого sidebar.

Основная логика:

```text
[ левая колонка 922 ] [ gap 18 ] [ правая колонка 350 ]
```

Левой колонке подчиняются:

- поиск и текущие production-фильтры;
- сортировка и добавление;
- таблица операций.

Правой колонке подчиняются:

- счета;
- расходы по категориям.

Верхняя строка совпадает с этими вертикалями:

- основные показатели занимают левую колонку;
- прогноз занимает правую колонку;
- промежуток между ними — тот же `18px`.

Компактная закреплённая панель показателей при прокрутке использует тот же общий `content-max` и выровнена с основной сеткой.

## Responsive

Desktop — основной целевой режим.

**Не скрывать правую колонку и прогноз только из-за недостатка ширины.** Пользователь знает, что эти данные существуют; их исчезновение будет восприниматься как поломка интерфейса.

При недостаточной ширине:

1. верхний блок прогноза переносится под основные показатели;
2. правая колонка переносится под таблицу;
3. порядок ниже таблицы: `Счета` → `Расходы по категориям`;
4. все блоки остаются доступны и видимы;
5. не использовать `display:none` для meaningful-контента;
6. не создавать горизонтальный scroll всей desktop-страницы, если содержимое можно корректно перестроить;
7. на ширине, где сама таблица 922px уже не помещается, сохранить существующий production mobile/tablet flow либо адаптировать его отдельно без потери данных.

Не ломай существующий mobile UX без необходимости. Не пытайся насильно переносить фиксированную desktop-таблицу на узкий экран.

# 7. Верхние показатели и прогноз

Показывать четыре основных значения периода:

- Расходы;

# 8. Поиск, текущие фильтры, сортировка и добавление

## 8.1. Граница текущей задачи

**Не переделывай семантику фильтров в рамках этой задачи.**

Фильтры будут отдельно перепроектированы позднее. До этого момента:

- сохранить текущую production-логику фильтрации;
- сохранить текущий production-набор фильтров;
- сохранить текущие правила применения, комбинирования, сброса и закрытия;
- не переносить из prototype экспериментальную модель `заменить / сочетать`;
- не переносить experimental chips и пояснения;
- не менять query-параметры, API и filter-state без необходимости.

## 8.2. Внешний вид toolbar

Несмотря на сохранение production-логики фильтров, **внешний вид toolbar должен максимально точно соответствовать prototype v34**:

- общий белый контейнер;
- размеры и внутренние отступы;
- border/radius/shadow;
- геометрия search area;
- положение существующих filter-controls;
- положение сортировки;
- split-кнопка `+ Добавить`;
- существующий production entry point импорта — только если он уже есть; его поведение не менять.

То есть сохраняется логика production-фильтров, но не их старый внешний вид.

Весь toolbar должен быть ровно `922px` и совпадать с шириной таблицы.

## 8.3. Поиск

Сохрани production-семантику поиска, но визуально реализуй его точно по prototype.

## 8.4. Сортировка

В production сортировки сейчас нет. Реализуй согласованный вариант из prototype:

- `Дата ↓` — default;
- `Сумма ↓`;
- `Сумма ↑`.

Поведение:

### `Дата ↓`

- операции сгруппированы по дням;
- дни идут от новых к старым;
- внутри дня сохраняется production-порядок либо стабильная сортировка по исходному времени/ID;
- показывается заголовок дня и сумма дня.

### `Сумма ↓` / `Сумма ↑`

- группировка по дням снимается;
- каждая операция становится отдельной строкой;
- появляется колонка `Дата`;
- сумма сортируется по абсолютному денежному значению операции в выбранной display/base currency согласно существующей production-логике;
- расходы по-прежнему показываются без минуса;
- поступления остаются зелёными и с плюсом;
- одинаковые суммы сортируются стабильно по дате, затем по ID.

При возвращении к `Дата ↓` дневная группировка восстанавливается.

# 9. Таблица операций

Production desktop использует **вариант А: отдельная колонка счёта**.

## 9.1. Ширина и сетка

Общая ширина таблицы — `922px`.

Desktop grid:

```css
grid-template-columns:
  24px   /* статус / внимание */
  132px  /* счёт */
  300px  /* назначение */
  244px  /* суммы */
  104px  /* категория */
  42px;  /* действия */

column-gap: 10px;
```

С учётом gaps, внутренних padding и border эта сетка должна помещаться ровно в левую колонку. Не оставлять заметный пустой «хвост» после кнопки действий.

## 9.2. Колонки

- индикатор внимания;
- счёт;
- назначение;
- сумма;
- категория;
- действия `⋮`.

Дата показывается в заголовке группы дня, а не повторяется в каждой строке основного режима.

## 9.3. Группировка по дням

Каждый день состоит из:

1. заголовка дня;
2. суммы дня;
3. списка операций.

Иерархия:

- сумма дня заметнее суммы отдельной операции;
- день воспринимается как группа;
- отдельные операции внутри группы остаются легко различимыми.

Целевые стили:

```text
Дата дня:          14px / 700
Сумма дня:         15px / 700
Описание:          13.5px / 500
Сумма операции:    13.5px / 500
Счёт:              12.5px / 400
Категория:          12.5px / 400
Локальная сумма:    12px / 400
Заголовки колонок: 11.5px / 600
```

Использовать стандартные веса `400 / 500 / 600 / 700`, а не промежуточные `550 / 650 / 750`, чтобы результат был предсказуемым на разных системах.

## 9.4. Разделители и плотность

- Между операциями внутри дня — ультралёгкая сплошная линия, ориентир `#f3f4f7`.
- После заголовка дня — более заметный divider `--line`.
- Не использовать пунктирные или точечные декоративные separators.
- Минимальная высота обычной строки — около `44px`.
- Вертикальные padding обычной строки — около `8px`.
- Обычная однострочная операция выровнена по вертикальному центру.
- Перевод с дополнительной строкой выровнен по верхнему краю и имеет высоту около `52px`.
- Hover строки сохраняется, ориентир `#faf9fc`.

## 9.5. Типы операций

### Расход

- сумма без минуса;
- нейтральный тёмный цвет.

### Поступление

- с плюсом;
- зелёное.

Общее правило `.main-amount` не должно перекрывать цвет income. Проверь specificity и визуальные regression-тесты.

### Перевод

- не считается расходом или поступлением аналитики;
- основная строка показывает исходный счёт;
- под суммой/рядом показывается направление `→` и счёт назначения;
- при разных валютах отображаются обе суммы;
- при одинаковой валюте не нужно показывать искусственный курс.

## 9.6. Действия строки

Кнопка `⋮`:

- обычная операция: `Править`, `Удалить`;
- связанный перевод: `Править`, `Разъединить`.

Не оставлять лишнее пространство справа от кнопки действий.

## 10.1. Счета

# 11. Форма расхода / поступления

## 11.0. Правило пользовательских текстов

Prototype v34 и отдельный файл `family-finance-approved-copy-v1.md` являются source of truth для пользовательских текстов.

Codex не должен самостоятельно:

- перефразировать подсказки;
- сокращать labels;
- придумывать error messages;
- заменять утверждённые названия кнопок;
- добавлять технические пояснения.

Если для edge-case нет утверждённого текста:

1. используй существующий production-текст, если он есть;
2. иначе пометь copy как product blocker/TODO;
3. не придумывай финальный текст скрытно.


Ручной ввод — основной сценарий.

## 11.1. Поля

- Тип;
- Счёт;
- Дата;
- Сумма;
- Описание;
- Категория — только для расхода;
- Заметка — опционально;
- Требует внимания — опционально.

Тип по умолчанию — `Расход`.

Порядок `Счёт` / `Дата` должен быть одинаковым в формах создания и редактирования.

## 11.2. Счёт и дата

При выборе/изменении даты:

- обновить список допустимых счетов;
- валидировать текущий выбранный счёт;
- не сохранять неконсистентную операцию.

Точные тексты:

Под полем даты:

> Дата влияет на список доступных счетов

Если выбранный счёт недоступен:

> Этот счёт недоступен на выбранную дату. Выберите другой счёт.

Не перефразировать эти сообщения.

## 11.3. Описание и автокатегоризация

Не вводить отдельный справочник «описаний».

Зафиксированное поведение подсказок:

- начинать поиск после ввода **2 символов**;
- debounce около **250 ms**;
- искать по истории операций за последние **12 месяцев**;
- показывать максимум **8 вариантов**;
- искать по нормализованному описанию без AI;
- ранжировать: точное начало строки → начало слова → прочие совпадения;
- внутри одинакового качества учитывать частоту и недавность;
- показывать название, категорию и количество использований в стиле prototype;
- выбор подсказки подставляет нормализованное описание;
- если category confidence достаточно высокий, подставить категорию и показать:
  `Подставлено автоматически`.

Почему 2 символа:

- 1 символ создаёт слишком много шума;
- 3 символа заставляют лишний раз перепечатывать короткие и часто повторяющиеся названия;
- для семейного объёма данных 2 символа не создают значимой нагрузки при debounce, limit и корректном index/query.

Почему 12 месяцев:

- сохраняются сезонные и ежегодные платежи;
- 6 месяцев могут потерять редкие, но повторяющиеся расходы;
- объём семейной истории остаётся небольшим.

Техническая реализация зависит от production-стека:

- если данные уже доступны на клиенте, допустим локальный индекс/cache;
- если используется backend-query, ограничить выборку user/family scope, датой и `LIMIT 8`;
- добавить/использовать index только после проверки существующей схемы;
- не загружать всю историю при каждом keystroke.

Не полагаться на отдельную массовую кнопку «применить автокатегории».

# 12. Переводы между своими счетами

Перевод — это две связанные операции, которые в интерфейсе редактируются как единая сущность.

## 12.1. Создание перевода

### Одинаковая валюта

- одно поле `Сумма`;
- не растягивать форму пустым вторым полем;
- принимаемая сумма синхронизирована со списанной.

### Разные валюты

- `Списано`;
- `Зачислено`;
- пользователь вводит обе фактические суммы;
- отдельный курс не нужен.

Дата — дата списания.

## 12.2. Поиск операции-напарника

Два сценария:

1. `Подходящие операции`;
2. `+ Создать операцию-напарника`.

Использовать переключаемые accordion-секции.

`Подходящие операции` открыта по умолчанию.

Карточка кандидата:

- сверху название счёта;
- ниже дата;
- справа сумма;
- без логотипа;
- показывать сумму исходной операции для сравнения;
- одинаковые отступы;
- дата выровнена по левому краю.

## 12.3. Редактирование связанного перевода

Ширина формы — около `660px`.

Перед полями нейтральный full-width banner:

> Связанные операции · изменения сохраняются для обеих сторон

Две компактные колонки:

- `Списано`;
- `Зачислено`.

Поведение:

- дата занимает левую половину;
- при одинаковой валюте сумма зачисления синхронизирована и read-only;
- при разных валютах сумма зачисления редактируется в том же месте;
- переключение валют не должно вызывать layout jump;
- доступно действие `Разъединить`.

## 12.4. Разъединение

Смысл действия:

> Эти две операции на самом деле не являются одним переводом.

После разъединения обе операции становятся обычными и начинают участвовать в аналитике как самостоятельные приход/расход в соответствии с их типом.

## 12.5. Комиссия

Комиссия должна учитываться отдельно от суммы перевода, чтобы:

- суммы перевода можно было сопоставить между счетами;
- комиссия учитывалась как настоящий расход;
- аналитика не теряла банковские комиссии.

Правило хранения:

- комиссия — обычная отдельная расходная операция;
- не прибавлять её к переводимой сумме;
- если production уже умеет связывать комиссию с переводом, сохрани эту связь;
- если отдельной модели/связи комиссии нет, **не создавай новую сущность только ради этого ТЗ**: сначала используй существующую модель обычной расходной операции;
- любое расширение схемы для специальной связи комиссии должно быть отдельно обосновано до реализации.

## 12.6. Заметка и внимание

При объединении операций:

- заметка и флаг внимания не теряются;
- если они добавлены на связанной стороне, при разъединении переходят на операцию списания согласно существующему согласованному правилу.

# 14. Accessibility и keyboard UX

Это не отдельный редизайн доступности всего приложения. Требования относятся только к компонентам, которые Codex создаёт или заметно переделывает в рамках этой задачи.

Минимально обязательно:

- у inputs/selects/buttons есть label или `aria-label`;
- раскрывающиеся панели корректно сообщают `aria-expanded`;
- menu и modal доступны с клавиатуры;
- новый modal удерживает focus внутри и возвращает его на кнопку открытия после закрытия;
- `Escape` закрывает верхний modal/menu, если это не приводит к потере несохранённых данных;
- focus визуально заметен;
- действие не должно быть доступно только через hover;
- зелёный цвет поступления дополняет плюс, поэтому смысл не передаётся одним цветом;
- новые вторичные тексты не должны выглядеть как disabled controls.

Не проводить широкую accessibility-миграцию старых экранов вне изменяемой области и не менять бизнес-логику ради формального соответствия чек-листу.

# 15. Data model и backend

## 15.1. Базовый принцип

**Не менять backend и схему БД по умолчанию.**

Сначала используй существующие production-сущности, API и связи. Prototype не является основанием для создания новых таблиц или полей.

Проверь, как сейчас представлены:

- account;
- transaction;
- category;
- transfer/link;
- note;
- attention flag;
- даты активности счёта;

Не предполагай наличие отдельной сущности `commission`.

## 15.2. Что допустимо без отдельного согласования

- адаптация frontend к существующим API;
- переиспользование существующей transaction-модели;
- хранение комиссии как обычного отдельного расхода;
- локальная нормализация знака и отображения при условии, что она не ломает существующие расчёты.

## 15.3. Когда требуется изменение backend/schema

Если без изменения модели невозможно корректно реализовать согласованное поведение:

1. остановись до migration;
2. опиши конкретный конфликт существующей схемы с UX;
3. предложи минимальный вариант;
4. перечисли затрагиваемые API, данные и обратную совместимость;
5. дождись отдельного решения либо явно пометь реализацию как blocked.

Не создавать заранее:

- специальную таблицу комиссий;
- дополнительные user-facing поля.


Если migration всё же отдельно одобрена:

- минимизируй её;
- обеспечь backward compatibility;
- добавь migration test;
- объясни, почему existing schema недостаточна.

1. Существующая production-логика фильтров не изменилась.

1. Закрытый счёт не доступен для новой даты.
2. Изменение даты инвалидирует неподходящий выбранный счёт.

2. Different-currency: две суммы без курса.

# 19. Важный принцип принятия решений

Не копируй каждую деталь прототипа механически.

При конфликте:

1. бизнес-правила и UX-инварианты из этого документа имеют высший приоритет;
2. затем — реальная production data model;
3. затем — визуальный prototype v34, кроме явно исключённых import-частей;
4. демонстрационный JavaScript prototype имеет самый низкий приоритет.

Не «улучшай» согласованные сценарии самостоятельно без необходимости. Если видишь реальную проблему, объясни её и предложи минимальный вариант, сохраняющий продуктовый смысл.
