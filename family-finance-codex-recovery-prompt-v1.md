Ты уже выполнил первую попытку реализации Phase 1, но результат не прошёл визуальную и функциональную приёмку.

Самостоятельно проведи полный аудит текущей ветки против:

Не считай текущую реализацию почти готовой. По видимому результату она является частичной смесью старого production UI и отдельных prototype-стилей.

# 3. Уже известные блокирующие дефекты

Следующие проблемы уже обнаружены пользователем. Исправь их независимо от результатов дополнительного аудита.

## 3.1. Смешаны старые и новые стили

Сейчас часть интерфейса стилизована по prototype, а часть осталась в старом production-виде.

Требование:

- все затронутые Phase 1-компоненты должны визуально соответствовать prototype v34;
- нельзя оставить старую форму/карточку/диаграмму и поменять только несколько цветов;
- исключить конфликтующие legacy selectors и точечные overrides;
- найти источник каждого legacy-style, а не маскировать его очередным `!important`.

Проверь отдельно:

- summary metrics;
- forecast;
- toolbar;
- таблицу;
- account card;
- category chart;
- operation modal;
- transfer forms;
- buttons;
- inputs/selects;
- overlays;
- typography.

## 3.2. Import UI находится внутри формы операции

В формах создания и редактирования операций сейчас отображается блок `Импорт выписки`.

Это запрещено в Phase 1.

Требование:

- полностью удалить import UI из:
  - новой расходной операции;
  - нового поступления;
  - редактирования расхода;
  - редактирования поступления;
  - transfer forms;
- не реализовывать новый import flow;
- если отдельный production entry point импорта уже существует, оставить его отдельно и не менять его поведение;
- форма операции заканчивается своими actions и не содержит import-секцию ниже.

## 3.3. Helper под датой показывается в edit-формах

Текст:

> Дата влияет на список доступных счетов

должен показываться в режиме создания операции.

В режиме редактирования уже существующего расхода или поступления этот постоянный helper не выводить.

При реальной ошибке выбранного счёта показывать утверждённый error:

> Этот счёт недоступен на выбранную дату. Выберите другой счёт.

Не заменять error постоянной общей подсказкой.

## 3.4. Неактивные счета показаны в dropdown

В новой операции dropdown сейчас показывает неактивные счета серым.

Требование:

- для выбранной даты показывать только счета, активные на эту дату;
- не выводить остальные неактивные счета disabled-строками;
- дата должна реально фильтровать список;
- при редактировании исторической операции текущий счёт доступен, если он был активен на дату этой операции;
- если выбранный счёт становится недоступным после изменения даты, показать invalid-state и потребовать другой выбор.

## 3.5. Диаграмма содержит только четыре категории

Четыре категории относятся только к компактной легенде под диаграммой.

Требование:

- donut строится по **всем категориям с ненулевыми расходами периода**;
- legend под диаграммой показывает четыре крупнейшие категории;
- остальные категории не должны исчезать из самой диаграммы;
- значения всех slices доступны через tooltip/interaction;
- не агрегировать остаток в `Остальное`, если этого уже не делает production-domain и это отдельно не согласовано;
- сумма slices должна совпадать с общей суммой расходов.

## 3.6. Выключенный прогноз оставляет пустое место

Когда прогноз выключен:

- блок четырёх основных показателей растягивается на всю ширину общей content-grid;
- он занимает область левой и правой колонок вместе;
- справа не остаётся пустая зарезервированная ячейка;
- нижняя сетка таблица + right column остаётся без изменений.

Проверить также sticky compact metrics при выключенном прогнозе.

## 3.7. Prototype-логика реализована не полностью

Не достаточно повторить внешний вид первого состояния.

Проверь, что реально работают:

- `Дата ↓` с дневной группировкой;
- `Сумма ↓/↑` без дневной группировки;
- дата в каждой строке режима сортировки по сумме;
- восстановление группировки после возврата к дате;
- same-currency transfer с одним полем суммы;
- different-currency transfer с двумя фактическими суммами;
- linking/unlinking;
- фильтрация счетов по дате;
- подсказки описаний после 2 символов;
- 12-месячный диапазон и limit 8;
- income с плюсом и зелёным;
- expense без минуса.

# 4. Визуальная приёмка не может быть декларативной

Не пиши «соответствует prototype» без доказательств.

Для каждого состояния из acceptance matrix предоставь:

- screenshot prototype;
- screenshot production после исправления;
- статус `PASS / FAIL`;
- краткое описание оставшегося допустимого расхождения;
- ссылку/путь на соответствующий visual test.

Критичные размеры и расположение блоков:

- sidebar: 72px;
- left column: 922px;
- right column: 350px;
- gap: 18px;
- toolbar равен table;
- forecast и right column выровнены;
- operation modal соответствует prototype по ширине, структуре и spacing.

Разница в несколько пикселей допустима только при технически объяснимой причине. Систематически другой layout — FAIL.

---

# 5. Требование к архитектуре исправления

Не наращивай слой случайных overrides.

Перед исправлением найди:

- legacy stylesheets, продолжающие влиять на новые компоненты;
- старые компоненты, которые нужно заменить, а не оборачивать;
- дублирующиеся design tokens;
- условные ветки, из-за которых prototype-style используется только в части состояний;
- старые import-фрагменты, встроенные в operation modal;
- data selectors, которые возвращают inactive accounts;
- chart transformation, которая обрезает dataset до top 4.

Предпочтительно:

- один источник design tokens;
- единая структура modal components;
- явные variants create/edit/expense/income;
- один selector активных на дату счетов;
- отдельно `chartDataAll` и `legendTop4`;
- feature-state forecast on/off без скрытой grid-column;
- тесты на бизнес-правила, а не только snapshots.

# 7. Когда можно снова показать результат пользователю

Только когда:

- все строки обязательной acceptance matrix имеют PASS;
- import отсутствует во всех operation/transfer forms;
- inactive accounts не видны;
- chart содержит все категории;
- forecast-off layout корректен;
- create/edit формы проверены отдельно;
- sorting behavior проверено;
- визуальные screenshots приложены;
- lint/typecheck/build проходят.

Пользователь не должен выполнять роль первичного тестировщика. Его проверка — финальная продуктовая приёмка уже системно проверенного результата.
